## What v1 does

Open Protocol on the dashboard. Three cards stack vertically:

1. **Biometrics** — sleep score, HRV, resting HR, stress. Pulled from Garmin Connect once per day via a small Python service running on Railway, or entered manually as a fallback.
2. **Today's macros** — kcal / protein / carbs / fat vs the day's goal, aggregated from the existing diary.
3. **Today's plan** — generated by Claude Sonnet 4.6: three meals (each with named items in grams plus computed macros), a workout (named blocks with sets / reps / intensity), and a recovery note that explicitly connects today's biometrics to today's plan.

Tap the chat FAB. Type "I only have 30 minutes today." Claude streams a response, calls a `regenerate_workout` tool with the new constraint, the server runs a focused sub-call to rebuild the workout JSON, persists it to `daily_briefing.workout`, and the card re-renders via Supabase Realtime. The whole loop is end-to-end in under 15 seconds.

## What I built this week

### New routes
- `POST /api/briefing/today` — assembles user_profile + today's biometrics + last 24h of macros + yesterday's workout, calls Claude with prompt caching on the system prompt + per-user profile, forces tool use of `emit_briefing` for guaranteed JSON, validates, upserts. Idempotent unless `?regenerate=1`.
- `POST /api/chat` — Server-Sent Events. Multi-turn agent loop, capped at 3 round-trips. Tool: `regenerate_workout`. Reuses the briefing's cache prefix so prompt cache hits stack across both endpoints.
- `POST /api/biometrics/sync` — calls the Railway Garmin service via a service-role-decrypted password; falls back to manual entry on PUT.
- `POST/GET/DELETE /api/integrations/garmin` — encrypts Garmin credentials with AES-256-GCM at rest.

### New schema (one migration `004_protocol_v1.sql`)
- `user_profile` (1:1 with auth.users) — freeform JSON for goals, dietary restrictions, equipment, weekly schedule.
- `biometrics_daily` (composite PK on `user_id, date`) — sleep score, HRV, RHR, stress, training load (acute/chronic). `source` = `garmin` or `manual`. `raw` jsonb stashes the full Garmin payload.
- `daily_briefing` (composite PK) — meals jsonb, workout jsonb, recovery note, model, `prompt_cache_hit`, `regenerated_at`. Realtime-enabled.
- `garmin_credentials` — encrypted password, no SELECT policy (server-side only via service role).

### New UI
- `/dashboard` — the demo-loop landing page. Three cards plus floating chat FAB. Subscribed to Realtime on `daily_briefing` and `biometrics_daily`.
- `/settings/integrations` — Garmin Connect connect / disconnect form.
- `components/coach/{BiometricsCard, MacrosCard, BriefingCard, ChatSlideOver, MessageBubble, ToolActivityChip}` — TS / Tailwind ports of HELIX's AI Dock components, sharing the same event vocabulary so the iOS app can plug into the same client state machine in v2.

### New infra
- `garmin-service/` — single-file FastAPI on Railway wrapping the community `garminconnect` library. Bearer-token-auth'd, stateless, never persists passwords. Why Railway and not Vercel: Garmin's multi-step auth + cold starts fight the 10s serverless timeout.

## What I'm proud of

- The system prompt with five worked examples (rest / hypertrophy / 5K-tempo / deload / cut) is the highest-leverage artifact in v1. Generic ChatGPT-y output is the dominant failure mode for AI fitness apps; the worked examples set tone + specificity + biometric-driven decision rules.
- Cache stacking: briefing + chat share the system prompt + per-user profile prefix. Cache hits compound across both endpoints for the same user/day.
- Realtime hookup: `regenerate_workout` and `swap_meal` both mutate `daily_briefing` server-side; the dashboard re-renders without any client-side refetch trigger. The only "magic" needed for the demo's "wow" moment.
- Manual-entry fallback for biometrics is wired from day one, not a stretch goal. If the Garmin lib breaks (which it will), the app keeps working.
- Parallel worktree agents (Wave 1: 4 agents, Wave 2: 3 agents) — the same dev-loop the class is teaching, running on the project itself. Each agent got a self-contained brief with explicit non-overlapping file ownership, so the seven branches merged into `main` with zero hand-fixed conflicts.

## Risk register and what's left

- **Garmin lib reliability.** Now on `garminconnect==0.3.3` (browser TLS impersonation + tokenstore session cache). Mitigated by the manual entry path either way. v2 considers an Apple Health bridge as a second path.
- **Prompt cache invalidation.** When the system prompt changes (which it will, as I tune the worked examples), all caches reset. Acceptable cost for a single user; v3 will think about this when there are multiple users.
- **Encrypted Garmin password storage.** AES-256-GCM with `GARMIN_ENC_KEY`. The threat model assumes a server compromise leaks ciphertext but not the env-var key. Documented in `garmin-service/README.md`. v2 may move this to Supabase Vault.
- **Tool-loop runaway.** Capped at 3 round-trips so a model loop can't burn tokens forever.
